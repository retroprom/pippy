Contribute to Pippy

The following is a list of tasks we have identified to improve or extend Pippy. Follow the links below for additional details and contact us if you wish to contribute.
  • Reducing object overhead
  • Reducing the code footprint
  • Generalizing import
  • Stackless Python
  • Stackless marshal
  • Interactive interface
  • Integration with PalmOS
  • Porting library modules
  • Minimizing startup/shutdown time
  • Incremental module imports

Reducing object overhead

Well known techniques can be applied to the object representations of the Python Virtual Machine to significantly reduce Python's consumption of heap space. For an excellent discussion of compact object formats consult Back to the Future:The Story of Squeak, A Practical Smalltalk Written in Itself which appeared in the Proceedings of the Conference on Object-Oriented Programming Languages and Systems (OOPSLA), Atlanta, Georgia, 1997.

Reducing the code footprint

A first-order reduction is mostly complete (long and unicode objects will be removed when we upgrade to Python 2.x). To reduce the size of Python without re-implementing key features will require eliminating non-essential features. For example dictionary objects have several convenience methods that are not strictly necessary. In the long run considerable space savings can be achieved by re-implementing core portions of the Python interpreter.

Generalizing import

Generalizing import to include import from Palm databases and URLs would improve the utility and ease of use of Pippy.

Stackless Python

When Pippy is upgraded to Python 2.x, it should be synchronized with the latest version of Stackless Python. The port should be straightforward and is critical to Python running safely on the Palm.

Stackless marshal

A pure Python marshal has been written to unmarshal .pyc files (one is included with Jython-2.0a2). In combination with Stackless Python, this provides a stackless solution. However, this module is nearly an order of magnitude slower than C-marshal; a more efficient C implementation is preferable.

Interactive interface

An interactive interface similar to that of LispMe is a desirable addition. The rudimentary interface provided with Pippy is modeled after the interactive portion of the LispMe interface.

PalmOS integration

Access to the PalmOS database functions and the PalmOS graphic resources from Python would put Python developers on an equal footing with other PalmOS application developers writing in C or C++. Several extension modules have been developed for our purposes and need additional work.

Porting library modules

Some Python library modules must be modified to accommodate the idiosyncrasies of the Palm platform. Examples of these are types.py and os.py.

Minimizing startup/shutdown overhead

Starting the PVM and cleanly shutting it down again can take tens of seconds. Reducing this dead time is desirable (though not critical).

Incremental module import

Often times one requires only a small number of functions or classes from a module. Significant space savings are possible if just the portion of a module that is required is loaded.
